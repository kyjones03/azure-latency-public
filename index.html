How it works

Azure Function Apps — one per regionA timer-triggered probe runs every 5 minutes inside each of the 9 U.S. source regions.
HTTPS probe to region-pinned Storage accountsEach probe connects to a dedicated Azure Blob Storage endpoint (<region>.blob.core.windows.net) in every other region. The storage accounts are accessed over Microsoft's backbone network via VNet service endpoints — not the public internet.
15 samples per pair, aggregated to P50For each source → destination pair, 15 sequential HTTPS requests are made. The probe captures DNS resolution time, TCP handshake duration, TLS negotiation, time-to-first-byte, and total HTTP round-trip. A warmup connection is discarded before sampling begins to avoid cold-start skew. The 15 samples are reduced to a P50 (median) for each phase.
Results published to Azure Event HubsOne ProbeEvent per source → destination pair is published to a shared Event Hub namespace using the Function App's managed identity — no secrets involved.
GitHub Actions reads and publishes the snapshotEvery 30 minutes, a workflow authenticates to Azure via OIDC federated credentials, reads the latest event per region pair from Event Hub, and writes data/snapshot.json. No browser credential is ever required.
This page reads the static snapshotOn load, the matrix fetches data/snapshot.json and renders the TCP RTT values. The metric shown is the median TCP handshake time — the purest measure of raw network latency between regions, without DNS or application overhead.

Baseline values for comparison are published by Microsoft at learn.microsoft.com/azure/networking/azure-network-latency.
